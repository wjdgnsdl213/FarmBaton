## 3. 막혀서 다음으로 넘긴 작업: PDF "3년 평균" 기능

### 사용자 요청
> 리포트에 작목별 소득데이터와 시세를 결합해 "이 농장의 3년 평균 예상
> 순소득과 인수 적정가"를 산출하는 데이터를 추가하는건 어떻게 생각하는지?

### 막힌 이유 (데이터 부재 — CLAUDE.md rule 8)
"3년 평균"을 만들 수 있다고 처음에 잘못 판단했었음. 실제로 코드를 다시
확인한 결과:

- `price_trend` 테이블(`db/migrations/002_core_schema.sql:85`)은
  **`crop_code`가 PRIMARY KEY** — 작목별로 딱 1행만 존재. 연도별 히스토리
  컬럼/테이블이 전혀 없음.
- ETL 스크립트 `etl/05_price_trend.py`는 KAMIS Open-API의
  `periodProductList`(일자 단위 소매가)를 호출해서 **"최근 30일 평균 vs
  1년 전 같은 30일 평균"** 비율 하나만 계산해 `price_trend.trend_index`에
  덮어쓰기(UPSERT)함. 스크립트 자체 주석(20번째 줄 부근)에 명시:
  > "There is no true '평년'(multi-year normal) endpoint in KAMIS
  > Open-API, so the year-over-year ratio is used as the closest available
  > proxy"
- 즉 **KAMIS Open-API 자체에 진짜 3개년(혹은 5개년) 평년 통계 엔드포인트가
  없고**, 우리 DB에도 과거 트렌드를 쌓아두는 히스토리 테이블이 없음. "3년
  평균"이라는 표현에 부합하는 데이터가 현재 파이프라인 어디에도 없음.

### 사용자 결정 (이 세션에서 확인됨)
이 문서를 작성해서 기록하고, 다음 작업자(Codex 포함)가 같은 결론에
도달할 수 있게 남기기로 함. **아직 아래 3가지 방향 중 어느 것을 선택할지는
결정되지 않은 상태** — 다음 작업자가 사용자에게 다시 확인하거나, 아래
권장안(옵션 A)으로 진행해도 됨.

### 검토된 선택지 (다음 작업자를 위한 정리)

**옵션 A — 지금 그릇으로 충분 (권장)**
`income_coef`(농산물소득조사 기반 10a당 평균소득, 원래 다년 평균 통계치)
× 현재 `trend_index`를 그대로 쓰고, PDF에 "이 시세 보정값은 최근 30일 vs
전년 동기 비교값이며, 5개년 평년 통계가 아님"이라는 주석만 추가. 추가
개발 거의 없음. **다만 이건 "3년 평균"이라는 사용자의 원래 요청을 그대로
충족하지는 못함** — 단어를 정직하게 바꾸는 선에서 타협.

**옵션 B — ETL로 매도 스냅샷을 쌓아가며 점진 축적**
`price_trend_history`류 타임시리즈 테이블을 새로 만들고, ETL을 주기적으로
돌려 매 실행 결과를 누적. 단 **6/30 대회 마감까지 3개년치를 쌓는 건
물리적으로 불가능** — 이번 대회 제출에는 못 쓰고, 장기적으로 서비스를
운영한다면 의미 있는 인프라.

**옵션 C — KAMIS의 다른 API/통계로 교체 가능성 조사**
지금 쓰는 `periodProductList`(일자 단위) 말고, KAMIS에 연도별 평균 통계를
주는 다른 엔드포인트가 있는지 재조사. (이번 세션에서는 조사 안 함 —
`etl/05_price_trend.py` 작성 시점에 이미 "없다"고 결론 내린 상태였으나,
재확인이 필요하면 KAMIS Open-API 문서를 다시 봐야 함.) 만약 있다면 가장
사용자 요청에 가까운 결과를 낼 수 있음.

### 다음 작업자가 할 일

## 4. "상담신청이 안 보인다" 버그 — 결론: 코드 버그 아니었음

청년농 매칭 리스트 상위에 이번 세션 테스트로 만든 더미 농장(주소
"충청북도 충주시 가주동 483")이 6건 중복으로 떠 있었음. 사용자가 매칭
리스트에서 상담을 신청했는데 그게 (사용자 본인 소유가 아닌) 테스트 계정
소유 농장으로 들어가서, 본인 대시보드 "상담 신청 보기"엔 안 보였던 것으로
추정. 직접 농가 등록→공개→청년농 등록→상담신청→대시보드 확인까지 전체
플로우를 운영 환경에서 재현했을 때 **정상 동작 확인함**. 더미 데이터는
이메일 패턴(`*_test_*@example.com`류)으로 식별해 전부 정리함. 앞으로
production에서 테스트할 때는 테스트 계정임을 알 수 있는 이메일을 쓰고,
테스트 후 정리하는 습관 필요.

## 6. 운영 환경 참고 (자주 까먹는 것들)

- **로컬과 production이 같은 Supabase DB를 공유함.** 로컬 backend를 띄워서
  테스트해도 운영 데이터에 직접 쓰임 — 테스트 후 꼭 정리할 것.
- **Railway 배포**: git push만으로 자동배포가 불안정함 — 수동
  `railway up --service backend --ci`를 우선 사용.
- **Vercel 배포**: Root Directory가 `.`로 설정돼 있어야 로컬 CLI 배포
  (`cd frontend && vercel --prod --yes`)가 정상 동작. `frontend`로 바꾸면
  경로가 두 번 겹쳐서 깨짐.
- **V-World 지오코딩이 Railway IP를 502로 차단**함 (datacenter IP 차단으로
  추정, V-World 측 인프라 이슈). 정적 폴백(`db/seed/geocode_fallback.csv`,
  9개 데모 주소)으로 우회 중. V-World 고객센터에 IP 화이트리스트 요청을
  넣었으나 응답 대기 중이고, Railway 트라이얼 플랜이라 IP가 안정적으로
  유지될지도 불확실.
- **로컬 backend 포트가 가끔 "유령 LISTEN" 상태**가 됨 (PID가 실제로는
  죽었는데 OS가 포트를 안 놔줌) — 그럴 땐 같은 포트로 재시도하지 말고 다른
  포트로 띄울 것.
